# Legacy Informatica PowerCenter Workflow: `wf_CRM_Lead_Opportunity_Extract`

## Purpose
Maintained legacy workflow that has run in production for several years, extracting lead and
sales-opportunity funnel data from the on-prem Oracle CRM database and loading it into the
enterprise data warehouse for sales/marketing funnel reporting. It is **not being replaced
outright** — this repo demonstrates maintaining it operationally while extending its output into
the new Delta Lake marketing platform (coexistence), per the JD's "Maintain, monitor, and
troubleshoot existing Informatica PowerCenter ETL workflows" and "Enhance and extend ETL
processes to support new data sources" responsibilities.

## Mapping overview
| Object | Type | Description |
|---|---|---|
| `s_m_CRM_Lead_Opportunity_Extract` | Session | Executes mapping `m_CRM_Lead_Opportunity_Extract` |
| `SQ_LEAD_OPPORTUNITY` | Source Qualifier | Incremental pull: `LAST_UPDATED_DT >= $$LAST_RUN_DATE` |
| `exp_STAGE_NORMALIZE` | Expression | Standardizes stage codes (NEW/MQL/SQL/OPP/WON/LOST) and trims free-text fields |
| `lkp_CAMPAIGN_DIM` | Lookup | Enriches each lead with originating campaign metadata from `CAMPAIGN_DIM` |
| `fil_VALID_RECORDS` | Filter | Drops records with null `LEAD_ID` or an unrecognized stage code |
| `upd_STRATEGY` | Update Strategy | Marks insert/update per CDC flag for target load |
| `TGT_LEAD_OPPORTUNITY` | Target | Enterprise DW `LEAD_OPPORTUNITY` table (append-only stage history) |

## Operational runbook (day-2 support, as required by the JD)
1. **Monitoring**: Workflow Monitor alerts on session failure via email task `eml_OnFailure`;
   on-call checks `PMSERVER` logs at `/informatica/server/infa_shared/SessLogs/`.
2. **Common failure modes observed**:
   - Oracle connection pool exhaustion during quarter-end sales push — mitigated by staggering
     the schedule away from the sales-ops quarter-close batch window.
   - `$$LAST_RUN_DATE` parameter file not refreshed — session picks up a stale/empty
     incremental window. Fix: parameter file is now regenerated by a pre-session command task
     rather than manually maintained.
   - Unmapped stage codes after a CRM configuration change added a new pipeline stage — added
     a reject-file review step and a data-quality allowed-values check (see
     governance/data_quality_rules.yaml) so new/unexpected stage values are flagged, not
     silently dropped.
3. **Root-cause troubleshooting approach**: check session log -> verify source row count vs.
   target row count in `MX views` (`REP_SESS_LOG`) -> replay from the last successful
   checkpoint using the session recovery option rather than a full reload.

## Coexistence / phased modernization plan
Rather than a risky "big bang" cutover, the lead/opportunity feed is extended (not replaced) to
also land in the cloud platform:

1. **Phase 1 (current, this repo)** — ADF pipeline `PL_Legacy_Informatica_Coexistence.json`
   reads the same Oracle source table PowerCenter reads, lands a copy into
   `bsi_bronze.crm_leads_opportunities` in ADLS Gen2/Delta. PowerCenter continues serving
   existing DW/BI sales-funnel reports unchanged — zero risk to current reporting.
2. **Phase 2** — Silver notebook joins `crm_leads_opportunities` into the BlueStacks Delta Lake
   funnel-conversion and campaign-ROI marts; new marketing analytics consumers are pointed at
   Delta instead of the legacy DW.
3. **Phase 3** — Once all downstream consumers of `TGT_LEAD_OPPORTUNITY` are migrated and
   validated against the Delta equivalent for 2+ reporting cycles, the PowerCenter workflow is
   formally decommissioned and reconciliation sign-off is documented for audit.

This staged approach reflects "day-to-day operational reliability" alongside "long-term
modernization" — the two things the JD explicitly says this role must balance.
