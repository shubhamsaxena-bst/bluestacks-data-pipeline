# Sample / Dummy Data

Generated by `generate_dummy_data.py` (stdlib only, fixed random seed — re-running reproduces
identical files). Use this to exercise the whole pipeline against real Azure resources before any
live marketing feeds are connected.

## What's here

```
sample_data/
├── generate_dummy_data.py
├── raw/                                   <- upload each dt=... folder into your ADLS Gen2 "raw" container
│   ├── ad_campaign_performance/dt=2026-07-04/part-0000.json   (211 rows total across 3 days)
│   ├── web_analytics_events/dt=2026-07-04/part-0000.json      (452 rows total across 3 days)
│   ├── crm_leads_opportunities/dt=2026-07-04/part-0000.json   (218 rows total across 3 days)
│   └── email_campaign_engagement/dt=2026-07-04/part-0000.csv  (239 rows total across 3 days)
└── seed/
    ├── dim_campaign.json          <- one-time seed for bsi_silver.dim_campaign (NOT a Bronze source)
    └── ctl_active_sources.json    <- rows for the ADF control table ctl.active_sources
```

Dates covered: `2026-07-04`, `2026-07-05`, `2026-07-06` — matching the `dt=<ingest_date>` partition
convention `01_bronze_ingest_marketing_data.py` expects at
`abfss://raw@<storage>.dfs.core.windows.net/<source_name>/dt=<ingest_date>/`.

## Design details worth knowing before you run the pipeline

- **Shared campaign_id namespace.** `CMP-1001`…`CMP-1070` (paid Google/Meta campaigns) plus
  `CMP-2001`/`2002`/`2003` (organic/direct/referral) are reused consistently across all four
  sources so the Silver-layer `dim_campaign` join in `02_silver_cleanse_enrich.py` has real
  matches — you should see actual `campaign_name`/`channel_type` values in Silver output, not just
  `"unattributed"`/`"unknown"` everywhere.
- **Intentional "unattributed" test case.** `CMP-1066`–`CMP-1070` are used in the raw data but
  deliberately left OUT of `seed/dim_campaign.json`. After running Silver, rows referencing these
  IDs should show `campaign_name = "unattributed"` / `channel_type = "unknown"` — this proves the
  `coalesce()` fallback in the enrichment join is working, not silently dropping rows.
- **Intentional bad records.** ~1-2% of rows in each source's first day (`dt=2026-07-04`) are missing
  a required field (e.g. `channel`, `event_id`, `lead_id`, `email_campaign_id`). Running
  `01_bronze_ingest_marketing_data.py` against this data should route those rows to the
  `bsi_bronze.<source>_quarantine` table via `data_quality_checks.quarantine_invalid_records()` —
  a good way to confirm the quarantine path works before real data ever hits it. The quarantine
  ratio stays under the 2% alert threshold in `governance/data_quality_rules.yaml`, so no alert
  should fire; you can raise `n_bad` in the generator if you want to test the alert path itself.
- **CRM funnel realism.** Each of the 70 leads progresses through a random-depth slice of
  `new → mql → sql → opportunity → closed_won/closed_lost`, with stage-change timestamps spread
  across all 3 days — enough for `funnel_conversion_by_stage` in the Gold notebook to show a
  believable drop-off curve.
- **Email funnel realism.** Each day, 50 recipients are sent one of 10 email campaigns, with
  realistic open (~40%) → click (~30% of opens) → unsubscribe (~3% of opens) → bounce (~5%)
  drop-off, so `email_engagement_summary` produces non-trivial open/CTR/unsubscribe rates.

## Regenerating or scaling up

Edit the constants at the top of `generate_dummy_data.py` (`N_PAID_CAMPAIGNS`, `events_per_day`,
`n_leads`, `RUN_DATES`, etc.) and re-run:

```bash
python3 sample_data/generate_dummy_data.py
```

## Loading order

1. Upload `sample_data/raw/*/dt=*/` into the ADLS Gen2 `raw` container, preserving the folder
   structure exactly (see `GETTING_STARTED_AZURE.md`).
2. Load `sample_data/seed/dim_campaign.json` into `bsi_silver.dim_campaign` **before** running any
   Silver notebook — the enrichment join reads this table and will error if it doesn't exist yet.
3. Load `sample_data/seed/ctl_active_sources.json` into your ADF control table (or substitute a
   static list if you're skipping the Azure SQL control table for now — see
   `GETTING_STARTED_AZURE.md`).
